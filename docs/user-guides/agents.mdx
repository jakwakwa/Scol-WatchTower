## What agents do

Agents are autonomous AI services or system workers that handle specialised tasks in the onboarding workflow.

### Active Agents (Powered by **Gemini 3 Flash**)

- **Validation Agent**: Verifies uploaded documents (bank statements, ID, proof of residence, and optional accountant letter file when provided) for authenticity, completeness, and cross-reference consistency. Receives base64-encoded file content from the `documents` table.
- **Risk Verification Agent**: Analyzes bank statements and accountant letters to assess FICA compliance and financial risk.
- **Quote Generator Agent**: Generates risk-adjusted pricing quotes based on applicant profiles and credit scores.
- **Sanctions Agent**: Screens applicants via OpenSanctions `/match/sanctions` as primary, then falls back to Firecrawl crawling across UN consolidated, OFAC, and FIC TFS lists if OpenSanctions fails. See [Firecrawl Check Contracts](/agent-contracts/firecrawl-check-contracts) for configuration details.
- **Reporter Agent**: Synthesis agent that combines ITC/sanctions check results with AI validation agent results into a single consolidated report for Risk Manager review. Runs only after both ITC and AI paths complete. Emits `reporter/analysis.completed` with combined flags and trust score.

### Planned Agents

The following agents are currently in development or planned for future releases:

- **Document Generator**: For legal agreements.
- **E-Signature Handler**: For managing DocuSign workflows.


## Agent dashboard

The **Agents** page serves as a control tower for your AI fleet, providing **real-time insights** directly from the database:

- **Total agents**: The complete registry of available and planned agents.
- **Active agents**: Agents currently processing live workflows.
- **Total callbacks**: The number of successful operations performed by agents (e.g., successful risk assessments).
- **Errors**: The number of failed operations or exceptions logged by agents.

> **Note**: Statistics are aggregated in real-time. "Callbacks" track successful `agent_callback` or task completion events, while "Errors" track logged workflow exceptions.

## Agent status cards

Each agent card displays:

- **Identity**: Agent Name and ID (e.g., `xt_risk_agent_v2`).
- **AI Model**: The specific AI model powering the agent (e.g., **Gemini 3 Flash**).
- **Status**:
    - **Active**: Operational and processing tasks.
    - **Inactive / Planned**: Not yet deployed or currently disabled.
    - **Error**: Experiencing critical failures.
- **Performance**:
    - **Last callback**: Timestamp of the most recent activity.
    - **Callbacks**: Total successful runs.
    - **Errors**: Total failed runs.

## Interpreting agent status

- **Active**: The agent is healthy, configured with a valid provider (e.g., Google Cloud Vertex AI), and successfully processing events.
- **Inactive**: The agent is a placeholder or has been manually disabled.
- **Error**: The agent has reported repeated failures. Check the specific error count for severity.

## Stage 3 agent architecture

During Stage 3 of the onboarding workflow, agents operate in a parallel architecture:

### Data sources

The Validation Agent receives documents from three sources:

| Source | Table | Content type |
| :--- | :--- | :--- |
| FICA document uploads | `documents` | Base64-encoded files (PDF, images) |
| Onboarding document uploads | `documentUploads` | Base64-encoded files |

### Parallel execution

Two independent paths run concurrently after all required documents and forms are received:

1. **ITC + Sanctions path**: performs credit checks and sanctions screening via external APIs
2. **AI Validation path**: runs the Validation Agent, Risk Agent, and Sanctions Agent in parallel via `performAggregatedAnalysis`

Both paths must complete before the Reporter Agent synthesizes the results. See [Workflows → Stage 3](/user-guides/workflows#stage-3-procurement--ai-verification) for the full flow.

## When to escalate

Escalate to engineering or operations if:

- An active agent (like Risk or Quote) shows a sudden spike in **Errors**.
- The **Last callback** time is significantly older than expected for a high-volume workflow.
- Critical agents stuck in **Error** state are blocking new applications.
