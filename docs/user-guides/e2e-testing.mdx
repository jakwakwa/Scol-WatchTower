## E2E Testing with Playwright

This project uses [Playwright](https://playwright.dev/) for End-to-End (E2E) testing.

There are two test modes:

1. **Standard suites** (dashboard/workflow/auth): use Clerk test-user auth from global setup.
2. **Local regression suite** (procurement fallback): local DB-only workflow scenario for targeted UI/API validation.

### Prerequisites

1.  **Bun**: This project uses [bun](https://bun.sh/) as the package manager.
2.  **Clerk Test User** (standard suites): You need a dedicated test user in your Clerk instance.
    *   **Email**: `account-manager+clerk_test@example.com`
    *   **Password**: *Ask the team for the test password*
3.  **Environment Variables**:
    *   Copy `.env` to `.env.test`.
    *   Ensure the following variables are set in `.env.test`:
        ```bash
        NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_test_...
        CLERK_SECRET_KEY=sk_test_...
        E2E_CLERK_USER_USERNAME=account-manager+clerk_test@example.com
        E2E_CLERK_USER_PASSWORD=...
        DATABASE_URL=postgresql://postgres:postgres@localhost:5432/controltower
        TEST_DATABASE_URL=postgresql://postgres:postgres@localhost:5433/controltower_test
        ```
    *   For local fallback regression tests, make sure `CLERK_SECRET_KEY` is available in your current shell (for session token bootstrap).

### Running Tests

Run all E2E tests:

```bash
bun run test:e2e
```

Run specific projects (e.g., only dashboard tests):

```bash
bun run test:e2e --project="dashboard tests"
```

Run in UI mode for debugging:

```bash
bun run test:e2e:ui
```

### Local Regression: Procurement Fallback

This regression is designed for local validation only. It simulates procurement automation failure and verifies:

- API fallback fields in `/api/risk-review`
- Risk Review manual-check UX
- Notifications fallback copy/label
- Workflow timeline/log messaging

If using Docker, start the test database first: `docker compose up -d test_db`

```bash
bun run test:db:reset
set -a; source .env; set +a
CI=true PW_WEB_SERVER_PORT=3012 TEST_DATABASE_URL="postgresql://postgres:postgres@localhost:5433/controltower_test" bun run test:e2e --project="workflow local fallback tests"
```

### UAT Banner Tests

The app shows a red "UAT SANDBOX — DO NOT ENTER REAL APPLICANT DATA OR PII" banner when `NEXT_PUBLIC_VERCEL_ENV=preview` (Vercel preview deployments). Two scripts verify this behavior:

- **`bun run test:e2e:banner-prod`** — Asserts the banner is *not* visible in production.
- **`bun run test:e2e:banner-dev`** — Asserts the banner *is* visible when simulating the preview environment.

### Test Environment

The `.env.test` file configures the test environment:

- **`DATABASE_URL`**: PostgreSQL connection string for the main database
- **`TEST_DATABASE_URL`**: PostgreSQL connection string for the test database (Playwright injects this when running `bun run test:e2e`)
- **`PW_WEB_SERVER_PORT`**: optional Playwright web server port override (defaults to `3000`)
- **`CLERK_SECRET_KEY`**: required by local fallback suite to create a test session token

### Playwright Config

Key settings in `playwright.config.ts`:

| Setting     | Value                                      |
| ----------- | ------------------------------------------ |
| Base URL    | `http://localhost:<PW_WEB_SERVER_PORT>`    |
| Browser     | Chromium (default)                         |
| Retries     | 2 on CI, 0 locally                         |
| Screenshots | On failure                                 |
| Traces      | On first retry                             |
| Web server  | `bun run dev -- --port <PW_WEB_SERVER_PORT>` |

## Writing Tests

### Using Fixtures

```typescript
import { test, expect } from "../../fixtures";
import { DashboardPage } from "../../pages/dashboard.page";

test("should load dashboard", async ({ authenticatedPage }) => {
  const dashboardPage = new DashboardPage(authenticatedPage);
  await dashboardPage.goto();
  await expect(authenticatedPage).toHaveURL(/.*dashboard/);
});
```

### Authentication Logic

*   **Standard projects**: `global.setup.ts` authenticates once with Clerk test-user credentials and writes storage state to `playwright/.clerk/user.json`.
*   **Reuse**: dashboard/workflow/authenticated projects reuse this saved state.
*   **Local fallback project** (`workflow local fallback tests`): bypasses global setup and generates a Clerk session token dynamically inside the test using `CLERK_SECRET_KEY`.

## CI/CD with GitHub Actions

The `E2E Tests` workflow runs on every push to `main` and on pull requests.

### GitHub Secrets

To make the tests work in CI, you **must** configure the following secrets in your GitHub Repository settings:

| Secret Name | Description |
| :--- | :--- |
| `CLERK_SECRET_KEY` | Your Clerk Secret Key (from Clerk Dashboard > API Keys) |
| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` | Your Clerk Publishable Key |
| `E2E_CLERK_USER_USERNAME` | The email of the Clerk test user |
| `E2E_CLERK_USER_PASSWORD` | The password of the Clerk test user |

  </Step>
  <Step title="Run tests in UI mode">
    ```bash
    bun run test:e2e:ui
    ```

    This opens an interactive interface where you can select and debug tests.
  </Step>

    After running tests, open `playwright-report/index.html` for detailed results.
  </Step>

### Workflow Configuration

The workflow is defined in `.github/workflows/e2e-tests.yml`. It:

1.  Checks out the code.
2.  Installs Bun and dependencies.
3.  Installs Playwright browsers.
4.  Resets the test database.
5.  Runs `bun run test:e2e` with the secrets injected as environment variables.
| Suite                          | Description                         |
| ------------------------------ | ----------------------------------- |
| `auth/login.spec.ts`           | Protected routes, public access     |
| `dashboard/navigation.spec.ts` | Dashboard load, sidebar, navigation |

## Troubleshooting

### Tests fail with auth errors

Standard suites:

- Ensure `.env.test` has valid `E2E_CLERK_USER_USERNAME` and `E2E_CLERK_USER_PASSWORD`.
- Ensure `playwright/.clerk/user.json` can be generated by global setup.

Local fallback suite:

- Ensure `CLERK_SECRET_KEY` is exported in the shell running Playwright.
- Ensure `TEST_DATABASE_URL` points to your PostgreSQL test database (e.g. `postgresql://postgres:postgres@localhost:5433/controltower_test`).

### Database state issues

Run `bun run test:db:reset` to get a fresh database with seed data.

### Port already in use

Override Playwright's web server port:

```bash
PW_WEB_SERVER_PORT=3012 bun run test:e2e
```

### Browser not installed

```bash
bun run playwright install chromium
```
