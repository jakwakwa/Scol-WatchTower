## E2E Testing with Playwright

This project uses [Playwright](https://playwright.dev/) for End-to-End (E2E) testing.

Testing is split into:

1. **Standard suites** (auth/dashboard/workflow): Clerk test-user flow through global setup.
2. **Local fallback regression** (workflow local fallback tests): local-only procurement failure scenario.

### Prerequisites

1.  **Bun**: This project uses [bun](https://bun.sh/) as the package manager.
2.  **Clerk Test User** (standard suites): You need a dedicated test user in your Clerk instance.
    *   **Email**: `account-manager+clerk_test@example.com`
    *   **Password**: *Ask the team for the test password*
3.  **Environment Variables**:
    *   Copy `.env` to `.env.test`.
    *   Ensure the following variables are set in `.env.test`:
        ```bash
        NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_test_...
        CLERK_SECRET_KEY=sk_test_...
        E2E_CLERK_USER_USERNAME=account-manager+clerk_test@example.com
        E2E_CLERK_USER_PASSWORD=...
        DATABASE_URL=postgresql://postgres:postgres@localhost:5432/controltower
        TEST_DATABASE_URL=postgresql://postgres:postgres@localhost:5433/controltower_test
        ```
    *   Test DB selection is automatic when running `bun run test:e2e`; Playwright injects `TEST_DATABASE_URL` for the test database.
    *   For local fallback regression tests, ensure `CLERK_SECRET_KEY` is available in your shell.

### Running Tests

Run all E2E tests:

```bash
bun run test:e2e
```

Run specific projects (e.g., only dashboard tests):

```bash
bun run test:e2e --project="dashboard tests"
```

Run in UI mode for debugging:

```bash
bun run test:e2e:ui
```

### Local-only Fallback Regression

This validates procurement-check fallback behavior end-to-end in a local DB scenario.

If using Docker, start the test database first:

```bash
docker compose up -d test_db
```

Then run the regression:

```bash
bun run test:db:reset
set -a; source .env; set +a
CI=true PW_WEB_SERVER_PORT=3012 TEST_DATABASE_URL="postgresql://postgres:postgres@localhost:5433/controltower_test" bun run test:e2e --project="workflow local fallback tests"
```

### Authentication Logic

*   **Standard suites**: `global.setup.ts` signs in once and stores auth state at `playwright/.clerk/user.json`.
*   **Local fallback suite**: skips global setup and generates a Clerk session token dynamically using `CLERK_SECRET_KEY`.
*   **Port override support**: `PW_WEB_SERVER_PORT` controls Playwright base URL and web server port (default `3000`).

## CI/CD with GitHub Actions

The `E2E Tests` workflow runs on every push to `main` and on pull requests.

### GitHub Secrets

To make the tests work in CI, you **must** configure the following secrets in your GitHub Repository settings:

| Secret Name | Description |
| :--- | :--- |
| `CLERK_SECRET_KEY` | Your Clerk Secret Key (from Clerk Dashboard > API Keys) |
| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` | Your Clerk Publishable Key |
| `E2E_CLERK_USER_USERNAME` | The email of the Clerk test user |
| `E2E_CLERK_USER_PASSWORD` | The password of the Clerk test user |

### Workflow Configuration

The workflow is defined in `.github/workflows/e2e-tests.yml`. It:

1.  Checks out the code.
2.  Installs Bun and dependencies.
3.  Installs Playwright browsers.
4.  Resets the test database.
5.  Runs `bun run test:e2e` with the secrets injected as environment variables.

### Notes

- The **local fallback regression project** is primarily for local debugging/validation.
- Keep CI focused on the standard suites unless you explicitly wire the local regression into CI.
